# 0007: 파이프라인 실행을 업로드 응답에서 분리 (async, still no queue)

- 상태: 승인됨
- 날짜: 2026-07-12
- 상위 문서: [0001](0001-decisions.md) (기술 스택 — "비동기 처리" 행), [0004](0004-design-system-handoff.md) (디자인 핸드오프 — 목록/상세 진행 상태 표시)

## 배경

[0001](0001-decisions.md)은 "큐 없이 단순 async/await, 업로드 요청 안에서 STT→LLM까지 순차 처리"로
결정했다. 실제 구현(`POST /api/meetings`)도 그대로 `await runPipeline(...)` 후 응답을 반환했다.

디자인 핸드오프([0004](0004-design-system-handoff.md))는 반대로 "Upload → status transitions
`uploaded → transcribing → summarizing → completed` (real app polls job status)"를 명시하고,
목록 페이지는 진행 중인 행에 프로그레스바를 그리며, 상세 페이지([Task 11])는 3초 폴링 파이프라인
카드를 갖는다 — 즉 업로드가 빠르게 끝나고 이후 상태 변화를 폴링으로 관찰하는 UX를 전제로 만들어졌다.

50MB(34분) 분량의 실제 녹음 파일로 처음 끝까지 테스트해보니 이 불일치가 드러났다: 업로드 모달의
"업로드 중…" 버튼이 STT+LLM 전체가 끝날 때까지(수 분) 비활성 상태로 멈춰 있었고, 상세 페이지로
이동했을 때는 이미 `completed` 상태라 폴링 파이프라인 카드를 실제로는 거의 볼 일이 없었다 —
Task 11에서 만든 기능이 사실상 죽은 코드나 다름없었다.

## 결정

`POST /api/meetings`는 파일 저장 + `meetings` 행 생성(`status: uploaded`)까지만 마치고 즉시
응답한다. STT→LLM 파이프라인은 Next.js의 `after()`(`next/server`)로 응답 전송 후 백그라운드에서
실행한다. 별도 큐/워커 프로세스는 여전히 도입하지 않는다 — Node 프로세스가 살아있는 동안
`after()` 콜백이 마저 실행되는 것을 이용할 뿐이다([0001](0001-decisions.md)의 "큐는 과함" 판단은
유지).

업로드 모달은 응답이 오는 즉시(파일 저장만 기다리면 되므로 수 초 내) 상세 페이지로 리다이렉트한다.
상세 페이지의 폴링 파이프라인 카드(Task 11)가 이제 실제로 `uploaded → transcribing →
summarizing → completed` 전이를 보여준다. 목록 페이지도 진행 중인 행이 있을 때만 3초 간격으로
현재 로드된 범위를 재조회해, 새로고침 없이 프로그레스바가 갱신되도록 했다(디자인 핸드오프의
"Processing rows show an indigo progress bar" 요구사항).

## 트레이드오프

- Node 프로세스가 응답 전송 직후 재시작/크래시되면 `after()` 콜백이 유실될 수 있다 — 로컬
  단일 프로세스 데모 규모에서는 감수 가능한 리스크. 실서비스 규모라면 별도 큐로 다시 승격해야 한다.
- 재시도(`retry`) 경로는 이미 비동기 실행을 전제로 만들어져 있어 변경 없음.

## 기각한 대안

- **BullMQ/Redis 큐**: 데모 규모에 과함, 인프라 추가 필요 — [0001](0001-decisions.md)의 판단 유지.
- **클라이언트 사이드 폴링만으로 서버는 여전히 동기 유지**: 업로드 요청 자체가 수 분간 열려있는
  문제(브라우저/프록시 타임아웃 리스크, 모달이 그 동안 아무 정보도 못 줌)가 그대로 남아 기각.
